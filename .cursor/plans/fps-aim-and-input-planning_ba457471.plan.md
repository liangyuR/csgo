## FPS 自动瞄准与按键触发规划

### 总体思路

- **目标**：在现有“目标 Pose 已能稳定检测”的基础上，实现“按住/按下某个键 → 自动把准星平滑拉向目标关键点（如头部）”的功能，并确保在 FPS 场景下自然、安全，不出现瞬移或抖动。
- **核心步骤**：
  - 从 Pose 检测结果中提取一个统一的“目标屏幕坐标”（2D 像素点）。
  - 计算该点相对屏幕中心（准星位置）的偏移向量。
  - 将偏移映射为一段时间内的鼠标移动（含加速度/减速度与最大速度限制）。
  - 通过按键逻辑控制“是否执行上述自动移动”。

### 一、目标点与坐标系设计

- **1.1 明确输入坐标系**
  - 若当前 Pose 检测基于屏幕截图/视频流：通常关键点坐标已经是像素级的图像坐标，(0,0) 在左上角，x 向右、y 向下，这与屏幕坐标一致，较易直接使用。
  - 若是从游戏内存/引擎读出的 3D 坐标：需要先通过视图/投影矩阵把 3D 世界坐标投影到屏幕 2D 像素坐标，再参与后续计算。
- **1.2 统一“目标点”定义**
  - 从 Pose 中选取一个关键点作为“瞄准点”，例如：
    - 头像：头中心（头部关键点上下/左右平均）
    - 身体：胸口或上半身重心
  - 对多关键点进行简单融合：
    - 例如 `target_x = mean(head_left.x, head_right.x)`，`target_y = mean(head_top.y, head_bottom.y)`。
- **1.3 屏幕中心定义**
  - 设定准星固定在屏幕中心 `(cx, cy) = (screen_width/2, screen_height/2)`。
  - 这一点是 FPS 通用约定：大部分时间不去动游戏内准星本身，只通过鼠标移动改变视角，使目标移动到 `(cx, cy)`。

### 二、从目标点到“期望鼠标偏移”的数学关系

- **2.1 像素偏移向量**
  - 计算当前帧：`dx_pix = target_x - cx`，`dy_pix = target_y - cy`。
  - 这个向量代表“若把准星移到目标上，需要怎样移动屏幕内容”。
- **2.2 鼠标移动与视角变化的映射**
  - 不同游戏/灵敏度下，“鼠标移动 1 像素”对应的视角变化不同，需要通过实验或用户配置给出一个比例系数：
    - 例如 `yaw_per_pixel`, `pitch_per_pixel`，或者等效地，一个“鼠标灵敏度系数”。
  - 计划中可以预留：
    - `sensitivity_scale`：将像素偏移缩放为实际鼠标移动量的系数。
- **2.3 FOV / 视角相关限制**
  - 设一个“最大可接受角度偏差/屏幕距离”，超出则不启用或直接放弃自动瞄准，避免从屏幕一侧瞬移到另一侧：
    - 像素层面：`sqrt(dx_pix^2 + dy_pix^2) < max_distance_pixels`。
  - 只对“已经比较接近准星”的目标进行辅助，减弱违和感和风险。

### 三、瞄准控制器设计（平滑与限制）

- **3.1 即时跳转 vs 平滑过渡**
  - 不采用“一帧内直接移动 full dx/dy”的方式，容易表现为“锁头瞬移”。
  - 采用**比例控制**/缓动方案：
    - 每一小步移动：`step_dx = k * dx_pix`，`step_dy = k * dy_pix`（0 < k < 1），再加上最小步长限制。
- **3.2 最大速度与加速度控制**
  - 设每帧/每次调用的**最大鼠标速度**：例如 `max_step_pixels_per_tick`。
  - 在计算 `step_dx/step_dy` 后，若向量长度超过最大速度，则做归一化再乘以最大速度，保证“再远也只会慢慢追过去”。
- **3.3 死区与停机条件**
  - 当 `sqrt(dx_pix^2 + dy_pix^2) < deadzone_pixels` 时，视为已经瞄准完成，不再移动鼠标：
    - 避免在目标附近来回抖动。
- **3.4 时间采样与循环频率**
  - 明确瞄准更新频率：
    - 例如每帧循环或每 5~10ms 调用一次控制函数。
  - 若 Pose 更新频率低于输入频率，考虑在两帧 Pose 之间插值/保持上一帧目标位置。

### 四、按键触发与状态机

- **4.1 触发模式选择**
  - 两种常见模式：
    - **按住生效**：按下某键（比如鼠标侧键或 `Alt`）时进入“自动瞄准模式”，松开即退出。
    - **切换模式**：按一次开启辅助，再按一次关闭（需额外 UI/日志提示）。
  - 计划倾向优先实现“按住生效”模式，行为更可预期。
- **4.2 状态机设计**
  - 定义几个基本状态：
    - `IDLE`：未触发/无目标。
    - `AIMING`：按键按下且有有效目标 → 进行平滑追踪。
    - `COOLDOWN`（可选）：刚停止瞄准后的短暂冷却，避免频繁切换。
  - 状态转移规划：
    - `IDLE -> AIMING`：按键按下 + 检测到有效目标。
    - `AIMING -> IDLE`：按键松开 或 目标丢失/距离超阈值。
    - `AIMING -> COOLDOWN`（可选）：目标突然消失或检测异常时，进入短冷却后回 `IDLE`。
- **4.3 按键检测实现位置**
  - 在主循环中统一读取按键状态，并将“是否开启自动瞄准”的布尔量传入瞄准控制器。
  - 避免在多个线程里分散读取按键状态，保持单一真源。

### 五、多目标与优先级策略

- **5.1 目标筛选**
  - 当 Pose 模块识别到多名人物时，需要选一名作为当前瞄准目标：
    - 优先最近**屏幕中心**的目标（`distance_to_center` 最小）。
    - 其次可考虑目标距离/大小（更近/更大更优先）。
- **5.2 目标锁定与切换策略**
  - 当正在 `AIMING` 某目标时：
    - 只要此目标仍在屏幕内且距离中心不超阈值，就保持锁定。
    - 只有当前目标消失或严重偏离时，才重新选择最近的其他目标。
  - 避免每帧都在不同目标之间跳来跳去。

### 六、与输入系统的集成方式

- **6.1 鼠标移动注入**
  - 在规划中抽象一个函数，如 `send_mouse_delta(dx, dy)`，具体实现可以是：
    - Windows：使用 `SendInput` / `mouse_event` 等 API，发送相对位移。
    - Python：使用 `pyautogui`、`pynput` 等库，但需注意后台/前台窗口行为。
  - 计划上保证：
    - 鼠标移动是**相对值**（delta），而不是直接设置绝对屏幕坐标，以适配 FPS 的视角机制。
- **6.2 按键触发注入（若需要自动开火等）**
  - 若未来扩展“自动开火”：
    - 同样抽象 `press_key(key)` / `release_key(key)` 接口，底层用系统 API 或现有输入模块实现。
- **6.3 线程与同步**
  - 建议规划为：
    - 检测线程：负责获取最新 Pose/目标点。
    - 控制线程：负责根据当前“目标点 + 按键状态”计算鼠标 delta 并发送输入。
  - 用线程安全队列/锁传递最近一帧的目标信息，避免互相阻塞。

### 七、异常处理与安全边界

- **7.1 目标丢失**
  - 当若干帧都没有检测到目标时：
    - 立即停止自动移动，状态回到 `IDLE`。
- **7.2 错误/极端值过滤**
  - 若 Pose 输出突然跳变（例如关键点飞到屏幕外）：
    - 在规划中加入简单的滤波/限幅：
      - 如 `clamp(target_x, margin, screen_width-margin)`。
- **7.3 手动紧急终止键**
  - 预留一个全局“紧急停止键”（例如 `F8`）用于立即禁用全部自动输入逻辑，便于调试与安全考虑。

### 八、调试与参数标定

- **8.1 可视化叠加层（Overlay）**
  - 在调试阶段，考虑在屏幕截图或调试窗口中叠加：
    - 屏幕中心十字。
    - 当前选中目标点、预计移动路径。
  - 帮助调整 deadzone、最大速度、sensitivity 等参数。
- **8.2 参数配置文件**
  - 将如下参数抽离为配置：
    - `sensitivity_scale`
    - `max_step_pixels_per_tick`
    - `deadzone_pixels`
    - `max_distance_pixels` / `max_angle_deg`
    - 各种按键绑定
  - 便于后续快速调参，无需改代码。
- **8.3 测试流程规划**
  - 制定一个简单的测试脚本：
    - 站在训练地图中，让目标在不同方向/距离移动，验证自动瞄准响应是否平滑、自然。
    - 在激烈战斗场景下，确保不会出现抖动、乱锁、拉扯感。

### 九、后续可选扩展

- 支持**预测瞄准**（根据目标速度预估未来位置）。
- 支持**不同关键点模式**（例如“锁头模式”“锁身模式”切换）。
- 与已有 UI/控制台集成，实时查看当前状态机、目标信息和参数值。
